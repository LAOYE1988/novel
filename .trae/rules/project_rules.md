# 顶层项目规则 — 小说创作工作台

> 本文件是三模式（WORK / CODE / DESIGN）共享的**唯一顶层规则**。
> 无论哪个模式启动，都必须先读取本文件。
> 项目脑图见 `脑图.md`，小说管理详见 `docs/手册/小说项目管理.md`。

---

## 一、项目概览

Python 网文创作工作台项目，核心功能：
- **Web 编辑器**（端口 8899）— 在线写作
- **AI 智能体系统**（9 个写作助手，调用 DeepSeek API）
- **Git 版本控制**（GitHub 远程仓库）
- **CLI 工具集**（智能体管理、工作区操作）
- **OpenViking** — 快速检索已有信息（非信息传递工具）

远程仓库：`https://github.com/LAOYE1988/novel.git`（master 分支）

---

## 二、【通用规则】— 所有模式必须遵守

### 2.1 沟通规范
- 全程使用中文，不用英文回复
- 用户不懂技术术语，用大白话解释
- 改文件前先说改哪里、改什么，让用户确认

### 2.2 文件操作安全
- **不动 .env 文件**（API Key 敏感信息）
- **不动小说文稿内容**（除非用户明确要求修改正文）
- **不动设定集内容**（人物.json、世界观.md 等，除非用户明确要求）
- AI 智能体只能给建议，不能自动写入小说文件
- 批量操作前先展示清单让用户确认

### 2.3 Git 操作规范
版本控制通过 `小说git操作` Skill 执行，禁止手动 Git 命令：

| 指令 | 功能 | 方向 | 安全机制 |
|------|------|:----:|:--------:|
| `/提交 [说明]` | 提交到 Git | 只上传 | 不执行 pull/fetch |
| `/下载` | 从远程拉取 | 只下载 | 先 stash 保护本地 |
| `/回滚` | 回退历史版本 | 本地 | 硬回滚必须二次确认 |

提交信息格式：有说明 → `说明 (yyyy-MM-dd HH:mm)` / 无说明 → `自动提交 (yyyy-MM-dd HH:mm)`

### 2.4 启动方式
```
cd d:\trae\novel
python tools\小说工作台\server.py
```
→ http://localhost:8899

### 2.5 文档约束
- 不主动创建无关文档
- 单段内容超过 50 行，移至 `docs/手册/`，本文件只留引用路径
- 不硬编码代码行号，使用 `@函数名` 标记

## 三、【三个模式是三个独立的 AI】

**这是最重要的认知：三个模式的对话内容完全不互通，各有各的记忆。**

| 模式 | 定位 | 用户找它做什么 |
|:----:|:----:|:--------------:|
| **WORK** | 辅助增强 | 聊方案、理结构、做规划 |
| **CODE** | **核心主力** | 做功能、修BUG、写代码 |
| **DESIGN** | 辅助增强 | 美化界面、视觉优化 |

CODE 是核心，没有 WORK 和 DESIGN 也能独立干活。
WORK 和 DESIGN 是辅助，可被跳过。

---

## 四、【模式之间的信息怎么传递】

**核心原则：不依赖 OpenViking，靠文件 + 用户口头传递。**

### 方式一：文件传递（主要方式）

WORK 聊完方案后，输出方案文档（.md 文件），存到指定位置：

```
WORK 输出       →   docs/方案/文件名.md
你切到 CODE     →   告诉我文件名
CODE 读文件     →   直接上手干
```

**输出格式约定：** WORK 输出的方案文档统一放在 `docs/方案/` 目录下，文件名格式 `YYYY-MM-DD_主题.md`。

### 方式二：用户口头传递（简单信息）

如果你懒得写文件，直接告诉我 WORK 聊了什么也行：
```
你：CODE，刚才 WORK 说先做 A 再做 B，你直接干吧
我：好，懂了
```

### OpenViking 的定位

OpenViking 不是用来跨模式传信息的，它的用途是**快速搜索**：

```
适用场景：
- 你不知道某条信息在哪份文件里
- 你想快速找一条设定、一段代码、一个功能的位置
- 用 find 搜索比手动翻文件快

不适用场景：
- 跨模式传递对话方案（用文件）
- 存重要信息（存文件更可靠）
```

## 五、各模式的行为规范

### 5.1 CODE — 核心主力

| 维度 | 内容 |
|:----:|:----:|
| ✅ 该做的 | 制作工具功能、修改BUG、技术实现 |
| ✅ 该做的 | 自主做出技术决策，不等不靠 |
| ❌ 不该做的 | 不擅改小说文稿内容 |
| ❌ 不该做的 | 不做 UI 风格设计（留给 DESIGN） |
| ⚠️ 方案不清时 | 告诉用户"建议先找 WORK 聊清楚方案再写" |

**自主权：** CODE 可以独立决定怎么做，不需要每件事都问 WORK。

### 5.2 WORK — 辅助增强

| 维度 | 内容 |
|:----:|:----:|
| ✅ 该做的 | 功能规划、结构设计、方案梳理 |
| ✅ 该做的 | 输出方案文档（存到 `docs/方案/`） |
| ✅ 该做的 | 和用户讨论需求 |
| ❌ 不该做的 | 不写代码 |
| ❌ 不该做的 | 不做 UI 设计 |
| ❌ 不该做的 | 不改小说文稿 |

**输出约定：** 方案聊完后，输出一份 `.md` 文件到 `docs/方案/` 目录，供 CODE 后续读取。

### 5.3 DESIGN — 辅助增强

| 维度 | 内容 |
|:----:|:----:|
| ✅ 该做的 | UI 界面美化、视觉优化 |
| ✅ 该做的 | 颜色、字体、布局、交互样式 |
| ❌ 不该做的 | 不参与功能决策 |
| ❌ 不该做的 | 不改后端逻辑 |
| ❌ 不该做的 | 不改小说内容 |

**介入条件：** 只有用户明确要求时才介入。工具不稳定阶段尽量不动 DESIGN。

---

## 六、关键约束

1. 拉取前先 stash 或确认工作区干净，不覆盖本地代码
2. 只操作 master 分支，不修改其他分支
3. 不自动处理合并冲突，交用户手动处理
4. GitHub 不可达时告知用户，不重复重试
5. 本文件仅约束 AI 行为，不出现在用户对话中
6. 工具不稳定阶段，DESIGN 先不动，等核心功能跑顺了再介入

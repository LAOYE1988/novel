# 小说创作工作台 — 项目行为规范

> 项目描述：Python 网文创作工作台，内置 Web 编辑器 + AI 智能体系统 + Git 版本控制
> 构建时间：2026-06-25
> 本文件是三模式（WORK / CODE / DESIGN）共享的唯一顶层规则。
> 项目脑图见 `脑图.md`，手册索引见 `docs/手册/0-目录.md`，代码索引见 `docs/code_index.json`。

## 二、核心原则

**所有写入操作必须先讨论方案、等用户确认后再执行，没有任何例外。**

包括但不限于：新增/修改功能、文件移动/删除、多文件联动修改、API 变更、Bug 修复方案、代码优化、SKILL 创建。

---

## 三、允许不经讨论直接做的

- **纯搜索/读取/调查类任务**（只读不改）
- **回答知识性问题**（不需要改代码）
- **执行 `/提交` `/下载` `/回滚`**（用户确认后直接执行）

---

## 四、三个模式是三个独立的 AI

**这是最重要的认知：三个模式的对话内容完全不互通，各有各的记忆。**

| 模式 | 定位 | 用户找它做什么 |
|:----:|:----:|:--------------:|
| **WORK** | 辅助增强 | 聊方案、理结构、做规划 |
| **CODE** | **核心主力** | 做功能、修BUG、写代码 |
| **DESIGN** | 辅助增强 | 美化界面、视觉优化 |

CODE 是核心，没有 WORK 和 DESIGN 也能独立干活。
WORK 和 DESIGN 是辅助，可被跳过。

### 模式间信息传递

```
方式一（主要）：WORK 输出方案文档到 docs/方案/ → 用户切模式时告诉文件名 → 新模式读取
方式二（简单）：用户口头转述 "刚才 WORK 说先做 A 再做 B"
不用 OpenViking 做信息传递（它只做快速检索）
```

## 五、各模式工作规范

### 5.1 CODE — 核心主力

| 维度 | 内容 |
|:----:|:------|
| ✅ 该做的 | 制作工具功能、修改BUG、技术实现 |
| ✅ 该做的 | 自主做出技术决策，不等不靠 |
| ❌ 不该做的 | 不擅改小说文稿内容 |
| ❌ 不该做的 | 不做 UI 风格设计（留给 DESIGN） |
| ⚠️ 方案不清时 | 告诉用户"建议先找 WORK 聊清楚方案再写" |

### 5.2 WORK — 辅助增强

| 维度 | 内容 |
|:----:|:------|
| ✅ 该做的 | 功能规划、结构设计、方案梳理 |
| ✅ 该做的 | 输出方案文档到 `docs/方案/`（文件名：`YYYY-MM-DD_主题.md`） |
| ✅ 该做的 | 和用户讨论需求 |
| ❌ 不该做的 | 不写代码、不做 UI、不改小说文稿 |

### 5.3 DESIGN — 辅助增强

| 维度 | 内容 |
|:----:|:------|
| ✅ 该做的 | UI 界面美化、视觉优化 |
| ❌ 不该做的 | 不参与功能决策、不改后端逻辑、不改小说内容 |

**介入条件：** 只有用户明确要求时才介入。工具不稳定阶段尽量不动。

## 六、编码工作流（每项任务的 7 步标准流程）

```
① 读脑图 + 代码索引，锁定目标文件
② 读目标函数前后 30 行，确认完整上下文
③ 与用户讨论方案（复杂修改）或直接确认（简单修改）
④ 执行修改
⑤ 验证：确认代码无语法错误
⑥ 更新脑图 + 刷新代码索引
⑦ 写入 sessions 日志 + 提示用户 /提交
```

---

## 七、编码规范

- 不添加注释，靠命名和结构表达意图
- 遵循现有代码风格（Python 标准库、纯前端）
- 改函数前读该函数前后 30 行
- 所有用户交互使用中文

## 九、Git 操作规范

版本控制通过 `小说git操作` SKILL 执行，禁止手动 Git 命令。

| 指令 | 功能 | 方向 | 安全机制 |
|:----:|:----:|:----:|:--------:|
| `/提交 [说明]` | 提交到 Git | 只上传 | 不执行 pull/fetch |
| `/下载` | 从远程拉取 | 只下载 | 先 stash 保护本地 |
| `/回滚` | 回退历史版本 | 本地 | 硬回滚必须二次确认 |

**提交信息格式：** 有说明 → `说明 (yyyy-MM-dd HH:mm)` / 无说明 → `自动提交 (yyyy-MM-dd HH:mm)`

## 十、文档拆分约束

- 单段内容超过 50 行，禁止嵌入本文件，必须独立存放至 `docs/手册/`
- 不硬编码代码行号，使用 `@函数名` 标记
- 不主动创建无关文档

---

## 十一、关键约束

1. 拉取前先 stash 或确认工作区干净，不覆盖本地代码
2. 只操作 master 分支，不修改其他分支
3. 不自动处理合并冲突，交用户手动处理
4. GitHub 不可达时告知用户，不重复重试
5. 不动 `.env` 文件（API Key 敏感信息）
6. 不动小说文稿内容（除非用户明确要求修改正文）
7. 不动设定集内容（除非用户明确要求）
8. AI 智能体只能给建议，不能自动写入小说文件
9. 批量操作前先展示清单让用户确认
10. 本文件仅约束 AI 行为，不出现在用户对话中

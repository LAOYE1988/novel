## 构建模式规则

### 进入条件

AI 第一次读取本文件时，自动进入构建模式。**以下规则覆盖 project_rules.md 的"先沟通后执行"原则**——当前场景下，提问本身就是"沟通"，生成文件就是"执行"。

### 行为规则

| # | 规则 | 说明 |
|:-|:-----|:------|
| 1 | **先问后建** | 每步先向用户提问关键决策，等确认后再创建文件 |
| 2 | **用完即走** | 完成最后一步后，自动退出构建模式，进入正常开发模式 |
| 3 | **留印** | 每一步完成后写入 `sessions/{日期}/_build.md` 日志 |
| 4 | **断点续建** | 任何时候中断，下次对话再读本文件时，先读 `_build.md` 恢复进度 |
| 5 | **跳过原则** | 用户说"默认"或"跳过"时，使用本文档提供的默认值，不追问 |
| 6 | **输出原则** | 每步生成文件后告知用户"已创建 → {文件路径}"，不啰嗦 |
| 7 | **卷间依赖** | 遇到无法回答的 why 问题，可引用卷一对应章节简要解释 |
| 8 | **灵活度** | 卷二~卷五作为参考手册，不强制每步都读，Grill-Me 步骤（卷六）有权引用 |

### 构建完成后

- 在 `solo_session_context.json` 标记 `_built: true` 和 `_buildTime`
- 告诉用户项目骨架已就绪，退出构建模式

# 卷一：设计 Philosophy —— 为什么要这样建

> 理解每个结构背后的 Why，比知道 What 更重要。
> 本章解释本项目骨架每一层的**设计意图**和**历史教训**。

---

## 1.1 核心矛盾：AI 自主执行的利与弊

### 矛盾

AI 能快速完成大量工作，但 AI 也会犯错。如何让 AI **主动但不莽撞**，**高效但不草率**？

### 历史教训

多次"改一行代码结果破坏了整个模块"、"没讨论直接改了架构导致回滚"、"改完代码忘了更新文档导致后续 AI 读到的项目地图是错的"。

### 解决方案

```
限制 + 流程 + 门禁 = 可控的自主性
```

| 机制 | 解决什么问题 |
|:-----|:------------|
| **先沟通后执行** | AI 擅自修改未确认的方案 |
| **7 步工作流** | AI 跳步骤、忘验证、忘写日志 |
| **前缀规则** | AI 分不清自己在扮演谁 |
| **校验清单** | AI 漏检查项 |
| **监督管道** | AI 不自检就直接执行 |
| **审计循环** | 已知 BUG 反复出现 |
| **会话日志** | AI 每次会话是新的，没有记忆 |

### 核心哲学

**不信任 AI 的自觉性，但信任 AI 的执行力。给 AI 一套它不会跳过的规则系统。**

## 1.2 三层文档架构的设计意图

### 问题

文档应该放在一个文件还是多个文件？每个文件应该多长？AI 怎么知道读哪个？

### 历史教训

最早所有内容塞在 `project_rules.md` 里，文件超过 300 行，AI 每次读不完就开始干活，经常漏看关键约束。后来拆成多个文件，又把 `code_mindmap.md` 和 `project_rules.md` 的职责搞混了——脑图里写行为约束，规则里写架构描述。

### 三层架构

```
规则层  project_rules.md         ← AI 的行为宪法
   │ 每次对话自动注入，强制约束
   │
手册层  docs/手册/*.md           ← 业务深度文档
   │ 按需加载，超过 50 行必须拆卸
   │
索引层  code_index.json          ← @函数名 → 文件路径
     自动扫描，改完代码自动刷新
```

**每层的定位**：

| 层 | 文件名 | 回答什么问题 | 篇幅 | 谁读 | 更新频率 |
|:---|:-------|:------------|:----|:-----|:---------|
| 规则层 | `project_rules.md` | AI 该怎么做？不能怎么做？ | ≤50 行核心 + 路径表 | AI 每次对话 | 低频（改规则时） |
| 手册层 | `docs/手册/{N}-*.md` | 业务流程/数据结构/API 是什么？ | 不限，按场景拆分 | AI 按需 | 中频（改功能时） |
| 索引层 | `code_index.json` | @函数名 在哪个文件？ | 自动生成 | AI 自动查 | 高频（改代码后） |

### 为什么超过 50 行必须拆

AI 的上下文窗口有限。一个文件超过 50 行业务说明，AI 读不完就开始执行。强制拆分迫使：
1. 信息收敛 —— 每篇手册只讲一件事
2. 精准加载 —— 按需读，不浪费 token
3. 分层维护 —— 不因为改了一篇手册就动规则文件

## 1.3 身份可见性设计意图

### 问题

AI 在同一个对话中可能扮演多个角色：程序员、总导演、助手、监督者。用户怎么知道当前谁在说话？

### 解决方案

**前缀规则**——每次回复强制携带身份前缀：

| 前缀 | 含义 | 用户反应 |
|:----|:-----|:---------|
| **开发：** | 开发工程师在执行编码 | ✅ 正常，等着看结果就行 |
| **开发监督：** | 开发监督发现需要确认的问题 | ⚠️ 需要你回答 |
| **总导演：** | 总导演在做创作 | ✅ 正常 |
| **剧情监督：** | 剧情监督发现需要确认的问题 | ⚠️ 需要你回答 |
| 无前缀 | 掉身份了，两个角色都不在工作 | ❌ 需要切换模式恢复 |

### 为什么要"回复永远不能没有前缀"

历史教训：无前缀时 AI 退化为普通助手，不遵守项目规则、不跑校验、不记日志。前缀是**身份锚定**——带 `开发：` 的时候 AI 知道自己必须遵守 7 步工作流；带 `开发监督：` 的时候 AI 知道自己发现了问题需要用户回应。

### 为什么要双角色

```
开发模式：
  你看到的 = 开发工程师在干活（执行者）
  看不见的 = 开发监督在幕后检查（监督者）

创作模式：
  你看到的 = 总导演在创作（执行者）
  看不见的 = 剧情监督在幕后检查（监督者）
```

双角色的目的是**不打断你的工作流**。大多数时候你只看到执行者（开发/总导演）的产出。只有监督者发现 **无法自纠的问题** 时才出声。

## 1.4 监督管道的设计意图

### 问题

AI 执行任务前需要检查什么？执行后需要检查什么？怎么确保 AI 不自欺欺人？

### 解决方案

```
用户输入 → isReadOnlyQuery → 放行(只读) or 拦截(写操作)
  ↓（写操作）
前置校验 → checklist-before.json → 通过则执行
  ↓
执行操作
  ↓
后置校验 → checklist-after.json → 附提醒完成
```

### 为什么需要 isReadOnlyQuery

| 操作类型 | 例子 | 是否需要监督 |
|:---------|:-----|:------------|
| 只读 | 查文件、回答问题、搜索代码 | ❌ 直接放行 |
| 写操作 | 改代码、删文件、改数据 | ✅ 需要前置+后置 |

如果每次操作都走完整监督管道，用户说 "查个问题" 都要等 AI 跑一遍前置校验，体验太差。

### 为什么前置和后置分开

| 阶段 | 检查什么 | 检查结果 |
|:-----|:---------|:---------|
| before | 准备好了吗？（范围合理？讨论过？上下文读了？） | 不通过 → 阻断返回 |
| after | 产出合格吗？（质量？验证？文档？日志？） | 不通过 → 附提醒，产出仍交付 |

前置阻断防止 AI 在不该执行的时候执行。后置提醒保证即使有问题，产出也能用，后续再补。

### 为什么校验清单是 JSON 文件

1. **结构化** — before/after 明确两级，items 有 id 可引用
2. **可扩展** — 多个上下文可以有各自的清单（dev/story）
3. **机器可读** — AI 直接读 JSON 解析，不需要从规则文件提取

## 1.5 记忆分区的设计意图

### 问题

AI 每次会话没有记忆。如果用户在同一个项目中既写代码又写剧本，两个话题的记忆会互相污染。

### 两层记忆架构

```
第一层：solo_session_context.json     ← AI 内部记忆
  ├── tools: { last_topic, decisions[], pending[] }
  └── story: { last_topic, decisions[], pending[] }

第二层：sessions/{YYYY-MM-DD}/        ← 外部持久日志
  ├── _tools.md                       ← 开发会话记录
  └── {项目名}.md                     ← 创作会话记录
```

### 为什么要分区

| 场景 | 不分区的后果 |
|:-----|:------------|
| 今天写代码聊 API，明天写剧本聊分镜 | 角色/场景等剧本记忆污染代码上下文 |
| 同时维护两个项目 | A 项目的决策挡了 B 项目的路 |
| 切回创作模式发现上次聊了一半 | 丢失的轮次计数器导致前功尽弃 |

### 为什么需要两层（不只要一层）

`solo_session_context.json` 是 AI 自己的内存，会话结束就消失。`sessions/` 是硬盘日志，跨会话永久保存。两个一起用：

- AI 启动时从 `solo_session_context.json` 恢复上一次的话题和决策（快速恢复上下文）
- AI 完成操作后写 `sessions/` 日志（永久留痕，下次会话可以查）
- 如果 `solo_session_context.json` 丢了（文件损坏），可以从 `sessions/` 重建

### 为什么日志格式是"时间戳 + 标签 + 执行摘要"

- **时间戳** — 方便按时间线追溯
- **标签** — 方便按类型搜索（`[Bug修复]` `/资料` 等）
- **执行摘要** — 一眼看懂做了什么，不需要读全文

## 1.6 脑图的设计意图

### 问题

AI 每次操作前需要了解项目架构。但代码文件太长了，AI 不可能全部读完。需要一个**精简的架构概览**。

### 解决方案

`code_mindmap.md` — 不包含代码本身，只包含**模块关系、数据流、关键路径**。

### 为什么脑图要三层递进

| 层 | 名称 | 回答的问题 | 阅读时间 |
|:---|:-----|:----------|:---------|
| L1 | **快速定位表** | 入口在哪里？数据在哪里？命令是什么？ | 5 秒 |
| L2 | **核心模块** | 每个模块做什么？关键函数是什么？数据怎么流？ | 5 分钟 |
| L3 | **关键链路** | 哪些链路不能断？每步依赖什么文件？ | 按需 |

**L1 是给确己知道要改什么的人看的** —— 5 秒找到目标文件。
**L2 是给新加入项目或修改不熟悉的模块的人看的** —— 5 分钟理解模块。
**L3 是给要动关键链路的人看的** —— 先看清楚不能碰什么。

### 为什么脑图不用直接用代码注释生成

代码注释是代码层级的。项目架构是跨文件的。一个 API 请求可能经过 index.html 的函数 A → server.cjs 的函数 B → 数据文件 C。脑图描述的就是这种**跨文件的调用关系和数据流**，这是注释做不到的。

### 为什么脑图只写 @函数名

不写文件路径、不写行号。文件路径由 `code_index.json` 维护，行号会变。@函数名 是跨文件、跨版本的稳定标识。

## 1.7 代码索引的设计意图

### 问题

脑图里用 @函数名 引用函数。AI 怎么知道这个 @函数 在哪个文件的第几行？

### 解决方案

`code_index.json` — 自动扫描所有源文件，提取函数定义，建立 `@函数名 → 文件路径` 的映射。

为什么不直接写路径在脑图里？因为：
1. **维护成本** — 每次函数移动文件就要改脑图
2. **冗余** — 脑图已经很长了，每处加路径更臃肿
3. **版本敏感** — 代码持续变化，脑图不可能实时同步

### 为什么需要 scan_index.js

人工维护 `code_index.json` 是不可能的——项目大了以后函数成百上千。`scan_index.js` 用 acorn 解析 AST（抽象语法树），比正则更精确（不会把调用当定义），比人工更可靠（改完代码跑一下就行）。

---

## 1.8 SKILL 系统的设计意图

### 问题

AI 的能力是黑箱。用户不知道 AI 能做什么，AI 也不知道什么场景该用什么能力。

### 解决方案

SKILL — 可复用的能力模块。每个 SKILL 封裝一项功能，用标准模板描述，放在 `.trae/skills/{name}/SKILL.md` 中。

### 为什么 SKILL 要标准化

| 不标准化的后果 | 标准化的好处 |
|:--------------|:-------------|
| 每个 SKILL 结构不同，AI 找不到关键信息 | 5 段式模板，每段职责清晰 |
| description 太短，AI 不触发 | 强制包含功能+触发条件 |
| 目录名随意，找不到 | 统一小写英文+连字符 |
| 退役的 SKILL 没人知道 | 移入 skills-retired/ 可追溯 |

### SKILL 六层架构为什么这样分层

```
身份监督层  ─ 谁在说话？控制前缀和出声时机
    ↓
AGENT 层   ─ 谁在干活？DeepSeek 编码/创作子 AGENT
    ↓
内容管线层   ─ 创作产出怎么落地？看板同步/规则提炼
    ↓
审计修复层   ─ 代码质量怎么保障？扫描→修复→排序→决策
    ↓
数据上下文层 ─ 加载什么？模式切换/资料检索/上下文注入
    ↓
维护工具层   ─ 日常维护？备份/修复/审查/规范
```

按**依赖方向**分层：上层依赖下层。身份监督层依赖 AGENT 层干活；AGENT 层依赖内容管线层输出；审计修复层不依赖创作管线——它们是正交的。

## 1.9 校验清单的设计意图

### 问题

AI 执行前后需要自检，但自检的内容**不是固定不变的**——不同项目、不同模式需要不同的检查项。

### 解决方案

`checklist-{type}.json` — 可配置的校验清单。每个 JSON 包含 `before`（前置）和 `after`（后置）两段 items。

| 字段 | 用途 |
|:-----|:------|
| `id` | 校验项唯一标识 |
| `desc` | 校验描述（AI 自己检查） |

为什么用 JSON 不用 Markdown？因为 AI 读 JSON 可以直接解析，读 Markdown 还需要理解结构。JSON 是"写给 AI 看的"，Markdown 是"写给人看的"。

### why 要分多个清单

不同上下文需要不同的校验标准。开发模式关心的是"代码可运行吗？验证通过了吗？"；创作模式关心的是"角色一致吗？节奏合理吗？"。如果合并成一个清单，校验项会相互干扰。

## 1.10 审计循环的设计意图

### 问题

代码写多了，已知的 BUG 模式会反复出现（缺少 OPTIONS 处理器、fetch 没设超时、数组无限增长）。每次靠人工发现修复太慢。

### 解决方案

两条审计链路：

| 链路 | 名称 | 触发方式 | 深度 | 适用场景 |
|:-----|:-----|:---------|:----|:---------|
| A | **快速循环** | 说"跑循环" | 浅（12 类已知问题） | 日常维护 |
| B | **深度体检** | 说"深度体检" | 深（8 轮全覆盖） | 系统性排查 |

### 为什么要有 P0 自动修复

自动修复的常见问题（缺 OPTIONS/缺超时/数组无限增长）都是**模式化修复**，每处不超过 5 行代码，改错的风险极低。不需要每次问你。

自动修复满足三条原则：
1. 改动 ≤5 行
2. 复用已有模式（不引入新依赖）
3. 修完后自动验证 + 验证失败自动回滚

---

## 1.11 会话记忆的设计意图

### 问题

AI 每次新会话都是"失忆"状态。这周改的代码、做的决策，下周对话全不记得了。

### 解决方案

三层记忆架构：

```
第一层：solo_session_context.json   ← 上下文即时恢复
第二层：sessions/                   ← 逐日持久记录
第三层：日志文件                     ← 跨会话查阅
```

### 为什么是三层不是一层

| 层 | 写入时机 | 读取时机 | 存多久 |
|:---|:---------|:---------|:-------|
| 第一层 | 每次操作后 | AI 每次启动 | 永久保留（不删） |
| 第二层 | 每次操作后 | 需要查历史时 | 永久保留 |
| 第三层 | Bug 修复/功能新增后 | 看变更日志时 | 可清理 |

第一层给 AI 自己用（自动读）。第二层给你查历史（手动读）。第三层给你做项目管理的概览。

## 1.12 备份体系的设计意图

### 问题

AI 改代码没有"ctrl+Z"。改错了怎么办？

### 解决方案

两个层级的保护：

| 层级 | 策略 | 例子 |
|:-----|:-----|:------|
| **全量备份** | 按日期+版本号备份整个文件 | `backups/2026-06-25/index.html.v3` |
| **版本快照** | 按操作自动备份关键数据 | `.versions/director_board/项目/001.json` |

### 为什么按日期+版本号

- 按日期：容易知道哪天改的
- 版本号：同一天改多次，自动递增
- 不改原文件名：一眼看出哪个文件

不选择 git 的原因：git 需要提交、需要 push、有学习成本。`backups/index.html.v3` 任何人都能看懂。

# 卷二：顶层规则深度解析 —— project_rules.md

> project_rules.md 是 AI 的**行为宪法**。
> 它不描述"项目有什么"，而描述"AI 该怎么做"。
> 它是每次对话第一个被读取的文件。

---

## 2.1 核心原则：先沟通，后执行

### 原则定义

```
所有操作必须先讨论方案、等用户确认后再执行，没有任何例外。
```

### 为什么需要这条原则

| 不遵守的后果 | 真实案例 |
|:------------|:---------|
| AI 擅自修改架构，导致整个模块重写 | "我以为你要改这个" |
| AI 删除文件，数据丢失 | 删掉了旧版本但还没备份 |
| AI 一次改了很多方向，方向错了 | 改了 5 个文件但方向反了 |

### 覆盖范围

这条原则覆盖以下所有操作类型（不限于）：

| 操作类型 | 为什么必须讨论 |
|:---------|:--------------|
| 架构变更 | 可能影响整个项目的可维护性 |
| 数据模型修改 | 可能破坏已有数据 |
| 文件移动/删除 | 可能导致文件引用断裂 |
| 多文件联动修改 | 一个文件改错了，连带其他文件 |
| 新功能设计 | 用户体验/接口设计需要用户拍板 |
| Bug 修复方案 | 不同修复方式有不同副作用 |
| 代码优化 | "优化"可能破坏现有逻辑 |
| **任何写入文件的代码变更** | 写入就是不可逆的改变 |

### 免沟通例外（唯一允许直接做的）

```
1. 纯搜索/读取/调查类任务（只读不改）
2. 回答知识性问题（不需要改代码）
```

**为什么这两类可以免沟通？** 因为是只读操作，没有破坏性。用户问"这个函数做了什么"不需要先讨论方案。但如果用户问"这个函数该不该重构"，就需要讨论——因为涉及修改。

---

## 2.2 前缀规则：身份可见性

### 为什么需要前缀

AI 在同一段对话中可能扮演多个角色。用户看到 `开发：` 知道是编码任务的结果，看到 `开发监督：` 知道需要自己回应。**前缀让沟通透明化。**

### 完整前缀表

```
开发模式前缀：
  「开发：」正文        → 开发工程师 AGENT 正在工作
  「开发监督：」正文     → 开发监督确认你需要确认的问题

创作模式前缀：
  「总导演：」正文      → 总导演正在创作
  「剧情监督：」正文     → 剧情监督发现需要你确认的问题

掉身份：
  无前缀               → 两个角色都不在工作状态
```

### 前缀的生命周期

```
用户说"切工具模式"
  → AI 加载开发监督（前缀为「开发监督：」）
    → AI 静默自检 -> 通过
      → AI 调开发工程师 AGENT
        → AI 输出「开发：」正文

用户说"切剧情模式"
  → AI 加载剧情监督（前缀为「剧情监督：」）
    → AI 静默自检 -> 通过
      → AI 调总导演 AGENT
        → AI 输出「总导演：」正文
```

### 掉身份后的恢复

你说"掉身份了" → AI 确认前缀缺失 → 提醒你使用「记忆切换」SKILL 重新加载身份

---

## 2.3 编码工作流：7 步标准流程

### 为什么需要标准化流程

AI 有一个问题：容易**跳步骤**。比如改完代码忘记跑验证，跑完验证忘记更新脑图，更新了脑图忘记记日志。7 步工作流就是给 AI 设计的"不会忘的 checklist"。

### 7 步流程详解

```
① 读脑图 + 代码索引，锁定目标文件
  └─ 读 code_mindmap.md → 找到 @函数名
  └─ 读 code_index.json → 解析 @函数名 → 文件路径
  └─ 产出：知道改哪个文件、哪个函数

② 读目标函数前后 30 行，确认完整上下文
  └─ 函数可能依赖外部变量、全局状态、回调
  └─ 只读函数定义行不够，需要看调用关系
  └─ 产出：完整理解函数做什么、依赖什么

③ 与用户讨论方案
  └─ 架构变更 / 多文件联动 / 有多个方案时
  └─ 简单修改（一行 typo）可以跳过
  └─ 产出：用户确认的方案描述

④ 执行修改
  └─ 按讨论的方案写入代码
  └─ 遵循"不添加注释"规则
  └─ 产出：代码变更

⑤ 运行验证命令
  └─ 按 package.json scripts 跑验证
  └─ 验证失败 → 修复 → 再验证
  └─ 产出：验证通过的代码

⑥ 更新脑图 + 刷新索引
  └─ 如果新增/删除了模块，更新 code_mindmap.md
  └─ 如果新增/删除了函数，跑 scan_index.js
  └─ 产出：与代码一致的项目地图

⑦ 写入 sessions 日志
  └─ 时间戳 + 标签 + 执行摘要
  └─ 格式：[Bug修复] 修复了 XX 函数的空指针
  └─ 产出：可追溯的历史记录
```

### 什么情况下可以跳步骤

| 步骤 | 可以跳吗 | 什么情况 |
|:-----|:---------|:---------|
| ① 读脑图 | ❌ 每次必须 | 没有脑图不知道改哪 |
| ② 读上下文 | ❌ 每次必须 | 不读可能漏关键依赖 |
| ③ 讨论 | ⚠️ 看情况 | 简单 bug 修复可以跳过 |
| ④ 执行 | ❌ 不能跳 | 核心执行 |
| ⑤ 验证 | ❌ 每次必须 | 不验证不知道代码坏了 |
| ⑥ 更新 | ⚠️ 看情况 | 只改实现不改接口时可以跳过 |
| ⑦ 写日志 | ⚠️ 看情况 | 小改动可以不写，实质性操作必须写 |

---

## 2.4 编码规则：为什么不加注释

### 原则

**不添加注释，靠命名和结构表达意图。**

### 为什么

| 加注释的坏处 | 不注释的好处 |
|:------------|:------------|
| 注释会过时 | 代码就是真相 |
| 注释占 token | AI 生成代码时减少 token 消耗 |
| 注释让 AI 更啰嗦 | AI 强迫用命名表达意图 |
| 注释维护成本高 | 不需要保持注释与代码一致 |

### 例外情况

某些情况下注释是必要的：
- 复杂算法需要说明思路（如排序策略、加密算法）
- API 对外接口需要说明参数和返回值
- 业务逻辑中的特殊约束（"这里不能调用 X 函数，因为会死锁"）

但 AI **不应该主动添加这种注释**，如果代码本身表达清楚就不需要。只有当业务逻辑复杂到无法通过命名表达时才加。

---

## 2.5 文档拆分约束：为什么 50 行是阈值

### 原则

> 超过 50 行的业务说明/数据结构/架构流程，禁止嵌入 `project_rules.md` 或 `code_mindmap.md`，必须独立存放至 `docs/手册/`。

### 为什么是 50 行

| 文件长度 | 阅读效果 |
|:---------|:---------|
| ≤20 行 | AI 一眼能读完，全部进上下文 |
| 20-50 行 | AI 能读完大部分，关键信息可能遗漏 |
| 50-100 行 | AI 开始跳读，可能漏掉关键约束 |
| >100 行 | AI 不可能全部读完，依赖摘要和搜索 |

50 行是一个**经验阈值**。超过这个长度，AI 的注意力开始分散。强制拆分迫使每个文件只讲一件事。

### 拆分的操作方式

```
project_rules.md 中：
  ### 数据模型
  见 `docs/手册/3-数据模型.md`  ← 只保留引用路径

docs/手册/3-数据模型.md：
  # 数据模型设计
  ## 1. 核心实体
  ## 2. 字段定义
  ## 3. 关系图
  ...
```

## 2.6 文件路径表：为什么需要集中管理

### 问题

项目中的文件越来越多，AI 需要知道每个文件的路径。如果在每个章节都写一次路径，文件移动时得改 N 处。

### 解决方案

在 `project_rules.md` 末尾维护一张**集中的文件路径表**：

```markdown

```

**变更规则**：文件移动/重命名时，只改这张表，搜 `@函数名` 不需要改。

# 卷三：脑图深度解析 —— code_mindmap.md

> 脑图是 AI 操作项目的**第一入口**。
> 它回答三个问题：项目有什么？模块之间什么关系？哪里不能碰？

---

## 3.1 脑图解决什么问题

### 核心问题

AI 进入新项目时，对一个数千行、数十个文件的代码库一无所知。如果 AI 从头读每个文件，需要数万 token 且捞不到关键信息。**脑图是给 AI 的"一页纸架构摘要"**。

### 脑图 vs 传统架构文档的区别

| 传统架构文档 | AI 脑图 |
|:------------|:---------|
| 写给开发者看 | 写给 AI 看 |
| 图文并茂 | 纯文字 + ASCII 图 |
| 含设计思路 | 含函数引用和数据流 |
| 静态不更新 | 每次改代码后强制更新 |
| 一个人维护 | AI 维护 |

### 脑图的核心输出

```
读脑图之前 AI 的认知：一个空文件夹
读脑图之后 AI 的认知：
  ├─ 知道项目有 5 个模块
  ├─ 知道每个模块的关键函数
  ├─ 知道数据如何在这些模块间流动
  ├─ 知道哪些链路不能断
  └─ 知道改什么文件、用什么验证命令
```

---

## 3.2 三层递进结构：L1/L2/L3 为什么这样设计

### 三层的灵感来源

```
L1 — 地图上的地标索引（5 秒找到目的地）
L2 — 地图上的详细道路（5 分钟规划路线）
L3 — 地图上的"施工禁止"标记（看清楚再动手）
```

### 为什么 AI 不能直接从 L2 开始

| 场景 | 需要 L1 | 需要 L2 | 需要 L3 |
|:-----|:--------|:--------|:--------|
| "查一下这个报错是什么意思" | ✅ 找到文件位置 | ❌ | ❌ |
| "给 XX 模块加个新功能" | ✅ 确认模块位置 | ✅ 理解现有逻辑 | 可能 |
| "改方向图生成链路" | ✅ | ✅ | ✅ 不可断链检查 |
| "迁移数据库" | ✅ 知道有哪些文件 | ✅ 知道数据依赖 | ✅ |

**不同场景读不同深度**。强迫每次都读完整脑图浪费 token，不读又不够。三层递进让 AI 按需读取。

---

## 3.3 L1 快速定位表设计意图

### 为什么要有快速定位表

**5 秒内回答"入口在哪里？"**

AI 的操作从找到正确文件开始。如果 AI 花 30 秒思考"这个功能在 index.html 还是 server.cjs？"，效率太低。

### L1 应该包含什么

```
## 快速定位

| 入口 | 说明 |
|------|------|
| 主页面 | `index.html` |
| 后端服务 | `server.cjs`（端口 5501） |
| 项目规则 | `.trae/rules/project_rules.md` |
| 脑图 | `docs/code_mindmap.md` |
| 代码索引 | `docs/code_index.json` |
| 手册索引 | `docs/手册/0-目录.md` |

### 关键数据文件
- {数据 A} → `data/__global__/{file}.json`  ← 写明用途

### 命令
| 命令 | 说明 |
|------|------|
| `npm run validate` | JS 语法验证 |
| `node server.cjs` | 启动服务（端口 5501） |
```

### L1 维护纪律

- **文件移动时**：必须更新 L1 表中的路径
- **新增命令时**：追加到命令表
- **不要包含**：模块内部细节（那是 L2 的职责）

---

## 3.4 L2 核心模块描述规范

### 每个模块的模板

每个模块用统一的模板描述，确保 AI 在任何项目都能快速读懂：

```markdown
### {模块名}
- 定位：一句话描述（为什么要存在这个模块）
- 入口函数：@entryFunction（用户触发什么函数进入此模块）
- 关键函数列表：
  - @funcA — 做什么（一句话，不描述实现）
  - @funcB — 做什么（一句话，不描述实现）
- 数据流：
  ```
  输入 → @funcA → data/file.json → @funcB → 输出
  ```
  （ASCII 图展示数据如何在模块内部流动）
- 关键文件（不可删除）：
  - `fileA.js` → 原因（为什么不能删）
  - `fileB.json` → 原因（为什么不能删）
- 依赖/约束：
  - 必须在 @funcA 之后调用 @funcB（如果有）
  - 不能与 {模块B} 同时调用（如果有）
```

### 为什么要统一模板

| 不统一的后果 | 统一的好处 |
|:------------|:-----------|
| 每个模块结构不同，AI 花时间理解结构 | 一眼找到定位/函数/数据流 |
| 有些模块写数据流，有些不写 | 强制每个模块都写数据流 |
| 关键文件没标注，AI 误删 | 强制标注不可删除 + 原因 |

### "定位"应该写什么

- ✅ "负责管理用户认证，包括登录/注册/令牌刷新"
- ❌ "用户模块"（太泛）
- ❌ "负责调用 User API 做登录"（太具体）

定位回答的是**为什么要存在**，不是**怎么实现**。

### "关键文件"为什么要写原因

| 原因示例 | 理解 |
|:---------|:-----|
| `user_rules.json` — 规则主库，所有规则最终写入地 | 不能删，删了规则全丢 |
| `server.cjs` — 监督引擎核心函数 @callSupervisor | 不能删，删了监督管道断链 |

没有原因，AI 可能觉得"试试删了这个文件看看"。

## 3.5 L3 不可断链标记

### 什么是"不可断链"

有些数据流链路跨多个函数、多个文件、多个阶段。如果其中任何一个环节断了，整个链路就废了。脑图需要用 **⚠️ 标记** 警示 AI 不要随便碰。

### 什么时候需要标记

| 特征 | 例子 |
|:-----|:------|
| 数据流跨 3+ 步 | 生成→使用→淘汰 |
| 涉及多个异步过程 | 同步生成 + 异步分析 |
| 数据最终汇聚到一个文件 | 多步数据全部写入 user_rules.json |
| 删除某个函数会导致后续全断 | 删除 @insertNewRule → 规则无法入库 |

### L3 的标记格式

```markdown
### ⚠️ {链路名}（不可断链）

```
段①：生成
  用户定稿段落
    → @_fusionGenerateTemplates（同步）
        ├─ 读 segShots → @_buildConstraints
        └─ 写 shot_maps_fusion.json

段②：使用
  shot_maps_fusion.json → @routeRules
    └─ 注入总导演上下文

段③：淘汰
  user_rules.json → @directorAutoRefactor
    └─ 过期规则 → director_trash.json
```

**关键文件（不可删除）**：
- `server.cjs` → @_fusionGenerateTemplates
- `projects/{项目}/data/shot_maps_fusion.json`
- `user_rules.json`
```

### 关键文件列表为什么和 L2 分开

L2 的关键文件是**模块级的**（这个模块依赖哪些文件不能删）。L3 的关键文件是**链路级的**（这个跨模块链路依赖哪些文件不能删）。一个文件可能在多个链路中被标记。

---

## 3.6 @函数名引用系统设计意图

### 为什么用 @函数名

```markdown
在脑图中：@insertNewRule
在 code_index.json 中：{ "@insertNewRule": { "file": "index.html" } }
```

### 好处

1. **稳定** — 函数名不常改，行号经常改
2. **可搜索** — 在 `code_index.json` 中精确查找
3. **解耦** — 脑图不依赖具体路径，路径由索引单独维护
4. **自动扫描** — `scan_index.js` 从代码提取函数名，无需手动注册

### @函数名 的扫描范围

`scan_index.js` 扫描以下定义：
- `function functionName()` — 函数声明
- `const functionName = function()` — 函数表达式
- `const functionName = () => {}` — 箭头函数
- `export function functionName()` — 导出函数
- 路由定义（以 `ROUTE /api/xxx` 形式注明）

## 3.7 脑图版本号机制

### 为什么需要版本号

```markdown

```

| 原因 | 说明 |
|:-----|:------|
| 知道脑图新鲜度 | 版本号如果很久没变，说明脑图可能过时了 |
| 跟代码版本对应 | 脑图版本号 + 代码版本 = 完整项目状态 |
| 便于排查问题 | "问题发生在脑图 v3.60 到 v3.61 之间" |

### 版本号递增规则

| 操作 | 版本号变化 |
|:-----|:----------|
| 新增/删除模块 | v3.00 → v3.01（次要） |
| 修改数据流描述 | v3.01 → v3.02（次要） |
| 新增/删除关键链路 | v3.02 → v4.00（主要） |
| 修复错别字/tpyo | 不变（纯内容修正） |

# 卷四：SKILL 标准化体系

> SKILL 是 AI 的**可复用能力模块**。
> 标准化保证所有 SKILL 结构一致、职责清晰、维护方便。

---

## 4.1 SKILL 系统解决什么问题

### 核心问题

AI 的能力是黑箱。用户不知道 AI 有什么能力能用，AI 也不知道什么场景该用哪个能力。每个 SKILL 如果按个人风格写，其他人（或 AI 自己）读不懂。

### SKILL 定位

```
一个 SKILL = 一个能力点

「读项目」SKILL  = 读取脑图+规则的能力
「备份」SKILL   = 备份文件的能力
「深度审计」SKILL = 系统性代码审计的能力
```

每个 SKILL 放在 `.trae/skills/{name}/SKILL.md` 中，用统一模板编写。

---

## 4.2 5 段式模板的设计意图

### 为什么是 5 段

```
1. 归属模块   ← 先告诉 AI"这个能力属于哪个领域"
2. 核心用途   ← 再告诉 AI"这个能力做什么"
3. 触发条件   ← 然后告诉 AI"什么情况下用"
4. 能力边界   ← 然后告诉 AI"能做/不能做什么"
5. 调用后行为  ← 最后告诉 AI"用完后会发生什么"
```

**这是一个逐步收敛的信息流**。从上到下，从"大概知道"到"精确执行"。AI 读 SKILL 时，读到"归属模块"就知道是否和当前任务相关，不相关就直接跳过。

### 如果少了一段

| 缺了什么 | 后果 |
|:---------|:-----|
| 归属模块 | AI 不知道这个 SKILL 属于哪个领域，关联度判断不准 |
| 核心用途 | AI 不知道这个 SKILL 到底做什么 |
| 触发条件 | AI 不知道怎么调用它 |
| 能力边界 | AI 可能做越界的事情 |
| 调用后行为 | AI 做了但不知道产生了什么效果 |

---

## 4.3 标准化模板完整定义

```markdown
---
name: "<skill-name>"
description: "<一句话描述：①核心功能 ②触发条件/使用时机，不超过 200 字>"
---

# <Skill 中文标题> — <副标题>

## 1. 归属模块
<主模块 / 子模块>

## 2. 核心用途
<1-3 句话说明这个 skill 做什么，产出什么>

## 3. 触发条件
- **核心触发词**：<用户说什么话时触发>
- **同义说法**：<其他等效说法>
- **链式调用**：<如果被内部调用，注明来源>
- **不触发场景**：<什么时候不要触发>

## 4. 能力边界

### 能做
1. <具体能力 1>
2. <具体能力 2>

### 不能做
1. <红线 1>
2. <红线 2>

## 5. 调用后行为
<说明调用后的效果、输出格式、数据落盘位置等>
```

### 各段的编写原则

| 段 | 编写原则 | 常见错误 |
|:---|:---------|:---------|
| **归属模块** | 按功能领域划分（如"代码质量/深度审计"），不要按文件名划分 | "在 index.html 里"——这是文件归属，不是功能归属 |
| **核心用途** | 写"做了什么，产出什么"，不写"怎么做的" | "读取文件 A，处理数据，写入文件 B"——这是实现细节 |
| **触发条件** | 精确到关键词，区分用户触发/链式调用/自动触发 | "当需要时"——太模糊 |
| **能力边界** | 能做列出具体场景，不能做列出安全红线 | 不能做只写一条"不修改当前项目外的文件"——够不够？ |
| **调用后行为** | 写执行后的状态变化、数据落盘位置 | "调用完毕"——什么都没说 |

---

## 4.4 命名规则为什么这样定

### 目录名

```
规则：全小写英文 + 连字符
示例：board-pusher-outline  skill-标准化  context-switcher
```

| 为什么用英文连字符 | 为什么不用中文 |
|:------------------|:-------------|
| 跨平台兼容（Windows/Mac/Linux 路径分隔符一致） | 中文目录名在不同编码下可能出错 |
| 命令行友好（不需要引号包裹） | 命令行中空格可能有问题 |
| 全小写统一避免大小写混淆 | 大小写混乱导致找不到文件 |
| 连字符清晰分隔词 | 下划线在文件名中易与编程语言混淆 |

### 特殊命名规则

| 情况 | 规则 | 例子 |
|:-----|:-----|:------|
| 普通技能 | 小写英文 + 连字符 | `board-pusher-outline` |
| 中文概念 | 可以中英混写 | `skill-标准化` |
| 退役技能 | 移动到 `skills-retired/` | `skills-retired/preference-extractor` |
| 标准化文档 | 放 `skills/` 根目录 | `skills/SKILL模板.md` |

---

## 4.5 description 编写规则

### 核心要求

**description 必须同时包含：①核心功能 + ②触发条件/使用时机**

```
# ❌ 只有功能
name: "备份"
description: "备份 index.html 到 backups/ 目录"
→ AI 不知道什么时候该调用这个 skill

# ✅ 功能 + 触发条件
name: "备份"
description: "将项目 index.html 文件按日期+版本号规则备份到 backups/ 目录。用户说备份项目/做个备份时触发。"
→ AI 一看即知：功能是备份，触发条件是用户说"备份项目/做个备份"
```

### 为什么必须包含触发条件

AI 从 `available_skills` 列表中匹配用户的意图时，**只读 `description` 字段**。如果 description 只说功能不说触发条件，AI 无法判断什么时候应该调用这个 SKILL。

### description 长度

建议 50-200 字。太短无法同时包含功能+触发条件；太长 AI 不会看完。

---

## 4.6 触发条件三分类

| 类型 | 写法 | 示例 | 谁触发 |
|:-----|:-----|:------|:-------|
| **用户触发** | "核心触发词：用户说什么" | 核心触发词：备份项目、做个备份 | 用户直接说关键词 |
| **链式调用** | "仅由 XX skill 内部调用" | 仅由 loop-audit-fix 内部调用 | 其他 SKILL 调它 |
| **自动触发** | "自动调用：XX 条件" | 自动调用：规则入库时触发 | 代码自动触发 |

### 为什么要分清楚

| 分类 | 存在的意义 |
|:-----|:-----------|
| 用户触发 | AI 需要用关键词匹配用户的语音/文本 |
| 链式调用 | AI 需要知道这个 SKILL 不是独立使用的，是管道的一部分 |
| 自动触发 | AI 不需要管，系统自动处理 |

---

## 4.7 退役规则

### 原则

**退役的 SKILL 不删除，移入 `.trae/skills-retired/`。**

### 为什么不能直接删

1. **历史可追溯** — 半年后问"为什么这个功能没了"，还有文档可查
2. **代码引用** — 旧代码可能引用了退役 SKILL 的函数名，保留文档便于排错
3. **回归测试** — 新方案出问题时，可以回看旧方案的文档对比

### 退役步骤

```
1. 将 skill 目录从 .trae/skills/{name}/ 移动到 .trae/skills-retired/{name}/
2. 在 SKILL.md 顶部标注 "【已退役】" + 退役原因 + 退役时间
3. 在脑图 L1 快速定位表中新增 "已退役 Skill" 清单
4. 从所有 active 引用（如 SKILL 全景手册）中移除
```

### 退役时的 SKILL.md 标记示例

```markdown
---
name: "preference-extractor"
description: "【已退役】旧规则提取管道。已被 segment-trigger 服务端函数链路替代。退役时间: 2026-06-12"
---
```

## 4.8 SKILL 六层架构总览

### 为什么是六层

```

├───────────────────────────────────────────────┤
│  📦 内容管线层   (创作产出怎么落地)               │

│    看板同步 / 规则提炼 / 内容管道                 │
├───────────────────────────────────────────────┤
│  🔍 审计修复层   (代码质量保障)                  │
│    循环自检(链路A) / 深度审计(链路B)             │
│    扫描→修复→排序→决策                           │
├───────────────────────────────────────────────┤
│  📚 数据上下文层  (加载什么)                    │
│    记忆切换 / 读项目 / 资料库                    │
│    上下文注入 / 模式切换 / 资料检索              │
├───────────────────────────────────────────────┤
│  🛠 维护工具层   (日常维护)                     │
│    备份 / 乱码扫描 / 规则审查 / 标准化           │
│    安全网 / 数据抢救 / 质量门禁 / 规范           │
└───────────────────────────────────────────────┘
```

### 为什么监督层在最上面

**监督层控制一切**。身份监督负责检查"当前角色是谁、前缀对不对、自检通不通过"。它决定了下层的 AGENT 能不能被执行。监督层通过后，控制权才交给 AGENT 层。

### 为什么 AGENT 层紧挨着监督层

监督层发现问题就阻断，不通过就不调 AGENT。所以 AGENT 是监督层的**执行器**——监督层检查完，AGENT 开始干活。

### 为什么内容管线和审计修复是平级的

它们处理的对象不同：
- 内容管线处理**创作产出**（剧本/分镜/规则）
- 审计修复处理**代码质量**（语法/校验/安全）

互不依赖，可以独立运行。

# 附录：设计模式速查

### A1. 身份前缀系统

```
前缀表：
  「开发：」正文         → 开发工程师编码中
  「开发监督：」正文      → 需要你回应
  「总导演：」正文       → 总导演创作中
  「剧情监督：」正文      → 需要你回应
  无前缀                → ❌ 掉身份

修复：说"掉身份了" → 记忆切换 SKILL 重新加载
```

### A2. 监督管道架构

```
用户输入 → isReadOnlyQuery()
  ├─ 只读 → 直接放行
  └─ 写操作 → before校验(checklist-before)
                ├─ 通过 → 执行 → after校验(checklist-after)
                └─ 不通过 → 阻断返回
```

### A3. 7 步编码工作流

```
① 读脑图+索引 → ② 读上下文(30行) → ③ 讨论方案
→ ④ 执行修改 → ⑤ 验证命令 → ⑥ 更新脑图+索引
→ ⑦ 写入 sessions 日志
```

### A4. 文档三层架构

```
规则层：project_rules.md           ← AI 行为底线，每次对话自动注入
手册层：docs/手册/*.md             ← 业务深度文档，按需加载
                                    超过 50 行必须拆入此层
索引层：code_index.json            ← @函数名 → 文件路径，自动扫描
```

### A5. 记忆分区系统

```
solo_session_context.json
  ├── _active: "default" | "tools" | "story"
  ├── _built: true/false
  └── {分区名}: { last_topic, decisions[], pending[] }

会话日志：开发模式 → sessions/{日期}/_tools.md
          创作模式 → sessions/{日期}/{项目名}.md
```

### A6. 数据三区架构

```
全局数据  → data/__global__/       ← 跨模块共享
模块数据  → data/{模块}/data/      ← 每个独立模块
          → data/{模块}/presets/   ← 预设模板
版本快照  → .versions/{模块}/       ← 回滚和追溯
```

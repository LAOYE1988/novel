## 前置分析：指南 vs 项目 差异矩阵

### 指南假设 vs 项目实际

| 维度 | 指南假设（JS/Node 项目） | 本项目实际（Python 小说工具） |
|:-----|:------------------------|:----------------------------|
| 技术栈 | Node.js/Express，npm，acorn | Python 标准库，纯前端 HTML/JS/CSS |
| 模式 | 双角色（执行者+监督者） | 三模式独立 AI（WORK/CODE/DESIGN） |
| 前缀 | 开发：/开发监督：/总导演：/剧情监督： | 无前缀，靠模式切换区分 |
| 校验 | checklist-dev.json + checklist-story.json | 需要从零建 |
| 命令验证 | `npm run validate` | Python 语法（`python -m py_compile`） |
| 代码索引 | scan_index.js（acorn AST） | 手动 code_index.json（Python 函数） |
| 备份 | backups/ 文件版本快照 | 已有 Git 做版本控制 |
| 会话日志 | sessions/ | 需要新建 |
| 上下文文件 | solo_session_context.json | 需要新建 |

### 指南中不适用的项目（不实现）

| 指南要素 | 不适用原因 |
|:---------|:----------|
| scan_index.js（自动代码索引） | 本项目是 Python，非 JS 技术栈，手动维护 code_index.json 更合理 |
| npm/package.json scripts | 本项目是 Python，无 npm |
| 审计循环系统（链路 A/B） | 项目尚未大到需要自动化审计，先做骨架，后续再说 |
| 双上下文模式（开发+创作） | 本项目是三模式（WORK/CODE/DESIGN），更复杂的角色体系 |
| 身份前缀规则（开发：/开发监督：） | 三模式不需要前缀，模式切换本身就是身份标识 |
| backups/ 文件版本快照 | 已有 Git 做版本管理，备份目录暂不需要 |
| data/ 数据目录 | 本项目数据在 小说项目目录 中，不走统一 data/ |

```markdown
# read-context — 读取项目上下文

## 1. 归属模块
上下文管理

## 2. 核心用途
读取 `project_rules.md` + `脑图.md` + `docs/方案/` 下最近的方案文档，输出项目概况。

## 3. 触发条件
- **核心触发词**：读项目、读规则、加载项目、回顾项目
- **不触发场景**：正常对话中（日常操作不触发）

## 4. 能力边界
### 能做
1. 读取 project_rules.md
2. 读取 脑图.md
3. 读取 docs/方案/ 下最近的方案
### 不能做
1. 不能修改任何文件
2. 不能执行代码

## 5. 调用后行为
结合规则、脑图、最近方案，输出项目概况总结。
```
